# 스펙: agent-coach MVP — 일일 에이전트 사용 분석·개선안 자동 기입

- 날짜: 2026-07-14 / 상태: 구현됨
- 관련: README.md(설계 개요), nohdol-harness `docs/specs/2026-07-14-agentsview-daemon-hook.md`(agentsview 연동 교훈), harness-install SKILL.md 3단계(설치 폴백)

## 배경

측정 도구(agentsview)는 세션·토큰·품질 신호를 로컬 SQLite에 인덱싱하지만, 측정 결과를 읽고 비효율 패턴을 찾아 다음 세션의 에이전트 행동을 바꾸는 지침으로 되돌리는 **피드백 루프**는 사람이 매일 하지 않는다. 사내 에이전트 사용자의 토큰 낭비를 구조적으로 줄이려면 이 루프의 자동화가 필요하다(2026-07-14 사용자 요청).

CLI 표면은 v0.38.1에서 실측했다: `usage daily --json`(일별 토큰·비용·모델 분해), `session list --json`(세션별 `total_output_tokens`·`peak_context_tokens`·`compaction_count`·`health_grade`·`quality_signals`), `daemon status`. agentsview는 2026년 도구로 모델 지식에 없다 — 연동 코드는 실측 출력 형태를 픽스처로 검증한다.

## 목표

- 하루 1회, 지난 측정(워터마크) 이후의 에이전트 사용을 분석해 비효율 발견 → 개선 지침 상위 3개를 `~/.claude/CLAUDE.md`·`~/.codex/AGENTS.md` 관리 블록에 기입하고, 전체 리포트를 로컬에 남기고, 데스크톱 알림으로 알린다.
- 비개발자가 명령 1회(`agentcoach install`)로 설치를 끝내고 이후 전 과정이 자동이게 한다.
- Windows·Ubuntu 1급 지원(macOS 병행), Go 단일 정적 바이너리(외부 런타임·라이브러리 의존 0).

## 비목표

- Claude Code hook 연동 — 설계 결정으로 배제(Codex 전용 사용자 누락, 비개발자 배포와 충돌).
- LLM 기반 분석·문구 생성 — v2. MVP 분석은 결정적 규칙만(토큰 절감 서비스가 분석에 토큰을 태우는 자기모순 금지).
- 세션 데이터의 머신 밖 전송, 팀 집계 대시보드 — v3(opt-in 별도 설계).
- agentsview 자체의 기능 보강(스키마 직접 읽기 등) — CLI 공개 표면만 쓴다.

## 요구사항

- R1 (수집): `agentsview usage daily --json`과 `session list --json`으로 워터마크(`state.json`) 이후 구간만 수집한다. agentsview 미설치·데몬 미실행·JSON 파싱 실패 등 어떤 경우에도 크래시 없이 사유를 리포트에 남기고 exit 0(fail-open).
- R2 (데몬 보장): 수집 전 `daemon status`로 실행 여부를 판정하되 **부정문 우선 배제**(정상 종료코드 + "not running"/"No … is running."/"stopped" 부재 + running 표기)로 판정한다 — 긍정 substring 단독 판정은 부정문을 실행 중으로 오판한다(하네스 2026-07-14 장애 교훈). 미실행이면 detached로 `daemon start`.
- R3 (분석): 결정적 규칙 6종 — ① 추세(직전 구간 대비 토큰·비용 증감) ② 대형 컨텍스트 세션(peak_context_tokens 임계 초과) ③ 컴팩션(mid_task_compaction 포함) ④ 프롬프트 위생(quality_signals 집계: short_prompt·duplicate_prompt·missing_verification·runaway_tool_loop) ⑤ 실패·재작업(tool_failure·tool_retry·edit_churn·health D/F) ⑥ 모델 믹스(프리미엄 모델 비용 점유율 + 소형 세션 비중). 임계값은 config로 조정 가능.
- R4 (지침 기입): 발견을 낭비 추정 점수로 정렬해 **상위 3개**만 지침 문장으로 변환하고, 대상 파일의 `<!-- agentcoach:begin -->`~`<!-- agentcoach:end -->` 관리 블록만 교체한다. **블록 전체 15줄 상한**(전역 지침은 매 세션 로드되므로 긴 코칭이 곧 토큰 낭비). 쓰기 전 백업(`~/.agentcoach/backups/`, 최근 7개 유지), 마커 훼손(begin만 존재·순서 역전·중복) 시 해당 파일 쓰기 포기하고 리포트에 기록. 대상 CLI 미사용자(`~/.claude`·`~/.codex` 디렉토리 부재)는 건너뛴다.
- R5 (리포트·알림): 전체 분석을 `~/.agentcoach/reports/YYYY-MM-DD.md`로 저장하고, 리포트 헤더에 지난 측정 대비 증감(%)을 명시한다. 데스크톱 알림(Windows toast / Ubuntu notify-send / macOS osascript)은 best-effort — 실패해도 무시.
- R6 (스케줄): `install`이 OS 스케줄러를 등록한다 — Windows는 schtasks 2건(DAILY 09:30 + ONLOGON), Ubuntu는 systemd user timer(OnCalendar 09:30, `Persistent=true`), macOS는 launchd(StartCalendarInterval + RunAtLoad). `analyze`는 하루 1회 멱등(state의 `last_run_date`가 오늘이면 스킵, `--force`로 무시)이라 중복 발화가 무해하다.
- R7 (설치·제거): `install`은 agentsview 부재 시 공식 설치 스크립트(macOS/Linux `install.sh`, Windows `install.ps1`)를 시도하고, 실패하면 GitHub Releases 수동 설치 경로를 안내하되 스케줄러 등록은 계속한다(analyze가 fail-open이므로). `uninstall`은 스케줄러 해제 + 관리 블록 제거를 수행하고, 데이터(`~/.agentcoach/`)는 `--purge`일 때만 삭제한다.
- R8 (출력 인코딩): stdout/stderr를 UTF-8(errors 대체)로 강제하지 않아도 되는 Go 문자열 처리만 쓰되, Windows 콘솔(cp949)에서 크래시하지 않아야 한다 — Go는 print에서 인코딩 예외가 없으므로 판정 로직에만 주의(R2).
- R9 (품질): `go vet`·`go test ./...` 통과, 크로스 컴파일 매트릭스(windows/amd64, linux/amd64, darwin/arm64) 빌드 성공. 외부 모듈 의존 0(표준 라이브러리만).

## 인터페이스 / 설계 개요

```
agentcoach install [--time HH:MM] [--no-agentsview]   # 설치·스케줄 등록·초기화
agentcoach analyze [--force] [--dry-run]              # 일일 분석 (스케줄러가 호출)
agentcoach report  [--date YYYY-MM-DD]                # 리포트 출력 (기본: 최신)
agentcoach uninstall [--purge]                        # 스케줄 해제·블록 제거
agentcoach version
```

패키지: `internal/config`(JSON 설정 — TOML 미채택: 표준 라이브러리 유지), `internal/state`(워터마크), `internal/agentsview`(CLI 래퍼), `internal/analyze`(규칙), `internal/advise`(지침 렌더), `internal/writeback`(관리 블록), `internal/report`, `internal/notify`, `internal/schedule`(OS별 등록). 데이터 흐름: state → agentsview → analyze → advise → writeback+report+notify → state 갱신.

- `~/.agentcoach/config.json`: 대상 파일 on/off, 임계값(peak context 등), 알림 on/off, 스케줄 시각.
- `~/.agentcoach/state.json`: `schema`, `last_run_date`, `watermark`(RFC3339), 직전 구간 합계(토큰·비용) — 추세 계산용.
- 관리 블록 형식(각 대상 파일 말미):

```markdown
<!-- agentcoach:begin v1 (자동 생성 — 직접 수정 금지, 매일 갱신) -->
## 토큰 효율 지침 (agent-coach, 2026-07-14)
- <지침 1> …
전체 리포트: ~/.agentcoach/reports/2026-07-14.md
<!-- agentcoach:end -->
```

## 완료 기준 (테스트 가능한 형태)

- [x] C1 (R4): 관리 블록 왕복 멱등 — 같은 내용 2회 쓰기 시 파일 불변, 다른 내용은 블록만 교체, 블록 외 내용 보존. 마커 훼손 3종(end 없음·역전·중복) → 쓰기 포기 + 원본 불변.
- [x] C2 (R4): 블록 전체 15줄 상한 — 발견이 많아도 넘지 않음.
- [x] C3 (R3): 실측 형태 픽스처(session list·usage daily JSON)에서 규칙 6종이 기대 발견을 산출.
- [x] C4 (R1): agentsview 부재 → analyze가 exit 0, 리포트에 미설치 안내 기록.
- [x] C5 (R2): daemon status 픽스처 — "No agentsview daemon is running."(rc 0)을 미실행으로 판정, running 상태는 재기동 안 함.
- [x] C6 (R6): OS별 스케줄 산출물(schtasks 인자·systemd unit·launchd plist) 내용이 기대와 일치(등록 실행은 OS 게이트).
- [x] C7 (R6): 같은 날 2회 analyze → 2회째 스킵, `--force`는 실행.
- [x] C8 (R9): `go vet`·`go test ./...` 통과 + 3종 크로스 컴파일 성공 + `go.mod`에 require 없음.
- [x] C9 (R7): uninstall이 관리 블록을 제거하고 블록 외 내용을 보존.
