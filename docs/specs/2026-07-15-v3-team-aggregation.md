# 스펙: agent-coach v3 — opt-in 팀 집계·벤치마크

- 날짜: 2026-07-15 / 상태: 초안
- 관련: `2026-07-14-mvp.md`(선행 — 로드맵 v3, 비목표 "머신 밖 전송"의 예외 설계), `2026-07-15-v2-personalized-coaching.md`(효과 측정 R6을 집계 지표로 재사용)

## 배경

MVP·v2는 개인 머신 안에서 닫힌다. 그러나 서비스의 최종 목표는 **사내 전체**의 토큰 비효율을 줄이는 것이고, 개인 리포트만으로는 ① 조직 차원에서 어디에 낭비가 몰리는지 볼 수 없고 ② 개인에게 "팀 평균 대비"라는 가장 강력한 동기 부여 신호를 줄 수 없다. 한편 세션 데이터에는 회사 코드·시크릿이 섞일 수 있어 원문 반출은 MVP부터 금지해 왔다 — 팀 집계는 이 가드레일을 완화하는 것이 아니라, **내용이 없는 집계 수치만** 명시적 동의 하에 보내는 별도 경로여야 한다.

## 목표

- 동의한 사용자의 일별 집계 지표가 사내망의 수집 서버에 쌓이고, 팀·조직 대시보드에서 추세와 낭비 분포를 볼 수 있다.
- 개인의 일일 리포트·알림에 "팀 평균 대비 ±%" 벤치마크가 표시된다.
- 전송되는 필드가 스키마에 화이트리스트로 고정되어, 클라이언트 코드 구조상 그 외 데이터는 전송이 불가능하다.
- 동의는 명시적·철회 가능하고, 미동의 사용자는 v2 이하와 동작이 완전히 같다.

## 비목표

- 세션 원문·프롬프트·파일 경로·git 정보의 전송 — 어떤 형태(해시 포함)로도 하지 않는다. 이유: 해시조차 사전 공격으로 역추적될 수 있고, 필요성이 증명된 적 없다.
- 개인 순위표(리더보드) — 감시 도구로 변질되는 순간 사용자가 설치를 회피해 측정 자체가 무너진다. 대시보드는 팀 단위 합계·분포까지만, 개인 식별 뷰는 본인 것만.
- 외부(사내망 밖) SaaS 호스팅 — 집계 수치라도 사외 반출은 별도 보안 검토 없이 하지 않는다.
- 실시간 스트리밍 — 일 1회 배치 push로 충분하다(analyze 주기와 동일).

## 요구사항

- R1 (opt-in): 전송은 `config.json`의 `team: {enabled, server_url, token}`이 전부 설정된 경우에만 활성화된다(기본 비활성). 활성화는 `agentcoach team join --server <url>` 명령으로만 가능하며, 이때 전송될 필드 목록 전문을 화면에 보여주고 y/N 동의를 받는다. `agentcoach team leave`는 즉시 전송을 중단한다. 이유: 설치 자동화(무인자 모드)에 동의가 섞이면 동의가 아니다 — join은 자동 모드에 포함하지 않는다.
- R2 (전송 스키마 화이트리스트): 전송 페이로드는 고정 타입 `type DailyMetrics struct`로만 구성된다 — 날짜, 익명 사용자 ID(R4), agent별 토큰 4계열 합계, 비용, 세션 수, 규칙별 발견 카운트·점수, health 등급 분포, 효과 측정 결과(improved/unchanged/worsened 카운트), 클라이언트 버전, 스키마 버전. 문자열 자유 필드는 스키마에 존재하지 않는다(v2 R2와 같은 타입 수준 차단). 스키마 변경은 이 스펙의 변경 이력에 기록돼야 한다.
- R3 (전송 동작): 일일 analyze 말미에 push한다(HTTPS POST, JSON). 실패는 로컬 큐(`~/.agentcoach/outbox/`)에 최대 14일 적재 후 다음 실행에서 재시도하고, 리포트 주석에 남긴다 — 서버 장애가 개인 코칭(로컬 경로)을 절대 막지 않는다(fail-open 계승). 14일 초과분은 버린다(무한 적재 방지).
- R4 (식별): 사용자 ID는 join 시 클라이언트가 생성하는 랜덤 UUID다. 서버는 UUID→표시명 매핑을 저장하지 않으며, 표시명이 필요한 조직 정책이면 join 시 사용자가 직접 입력한 별칭만 쓴다(사번·이메일 자동 수집 금지). 이유: 벤치마크에는 구분만 필요하지 신원은 필요 없다 — 신원 수집은 리더보드화(비목표)의 입구다.
- R5 (수집 서버): 단일 바이너리 `agentcoach-server`(같은 저장소 `cmd/agentcoach-server/`, Go+SQLite, 외부 의존 최소)로 배포한다. 엔드포인트 — `POST /v1/metrics`(bearer token 인증, 스키마 검증 후 저장), `GET /v1/benchmark?date=`(팀 평균·중앙값·분포 반환), `GET /`(대시보드 HTML). 스키마 밖 필드가 온 요청은 400으로 거부한다(관대한 수신은 화이트리스트를 무력화한다).
- R6 (대시보드): 서버 내장 웹 UI — 조직 합계 추세(토큰·비용), 규칙별 발견 분포, 참여율. 개인 상세는 본인 UUID 입력 시에만(비목표의 리더보드 방지). 별도 프런트엔드 프레임워크 없이 서버가 렌더한다(배포 단순성).
- R7 (개인 역반영): analyze가 push 후 `GET /v1/benchmark`를 조회해 "팀 평균 대비 토큰 ±%"를 일일 리포트 헤더와 알림에 추가한다. 조회 실패 시 벤치마크 줄만 생략(fail-open).
- R8 (보존·삭제): 서버 데이터 보존 기간은 서버 config로 강제하고(기본 180일), `agentcoach team leave --erase`는 서버에 본인 UUID 데이터 삭제를 요청한다(`DELETE /v1/metrics?user=`, 본인 token 검증). 이유: 철회 가능한 동의에는 삭제권이 포함돼야 사내 개인정보 정책과 정합한다.

## 인터페이스 / 설계 개요

```
cmd/agentcoach-server/     — 수집 서버 진입점
internal/team/             — 클라이언트: DailyMetrics 조립(화이트리스트 타입), push, outbox 재시도, benchmark 조회
internal/server/           — 저장(SQLite)·검증·벤치마크 계산·대시보드 렌더
```

- 클라이언트 CLI 추가: `agentcoach team join|leave|status` (leave에 `--erase` 옵션).
- 전송 경계면: `POST /v1/metrics` body = `{schema: 1, user: "<uuid>", date: "YYYY-MM-DD", metrics: DailyMetrics}` — 이 스키마가 클라이언트·서버 공용 Go 타입 하나로 정의된다(이중 정의 금지, 어긋남 방지).
- 서버 배포: 사내 VM/컨테이너 1대 + SQLite 파일. k8s 매니페스트는 사내 배포 결정 후 별도(release 스킬 경로).

## 완료 기준 (테스트 가능한 형태)

- [ ] C1 (R1): team 설정 없는 기본 상태에서 네트워크 push 코드 경로가 호출되지 않는다. `team join`은 필드 목록을 출력하고 N 응답 시 설정을 저장하지 않는다. `team leave` 후 push가 중단된다.
- [ ] C2 (R2): `DailyMetrics`에 문자열 자유 필드가 없고, 직렬화된 페이로드에 픽스처의 세션 원문·프로젝트명·파일 경로 조각이 포함되지 않는다.
- [ ] C3 (R3): 서버 불달 픽스처에서 analyze가 exit 0, outbox에 적재, 다음 실행에서 재전송 성공 시 outbox가 비워진다. 15일 된 outbox 항목은 폐기된다.
- [ ] C4 (R4): join이 생성한 UUID가 랜덤이고(2회 join 시 상이), 페이로드에 사번·이메일·머신명이 없다.
- [ ] C5 (R5): 서버가 스키마 밖 필드 포함 요청을 400으로 거부하고, 유효 요청을 저장 후 benchmark 조회에 반영한다. 잘못된 token은 401.
- [ ] C6 (R7): benchmark 응답 픽스처에서 일일 리포트 헤더에 "팀 평균 대비" 줄이 추가되고, 조회 실패 시 그 줄만 없이 리포트가 정상 생성된다.
- [ ] C7 (R8): `team leave --erase` 후 서버에 해당 UUID 데이터가 0건이다. 보존 기간 경과 데이터가 서버 정리 작업에서 삭제된다.
- [ ] C8: 클라이언트 크로스 컴파일 매트릭스(MVP R9)에 서버 바이너리(linux/amd64)가 추가되어도 클라이언트 외부 의존 0이 유지된다.

## 미해결 질문

- 서버 호스팅 위치·운영 주체(사내 VM? 팀 공용 장비?)와 HTTPS 인증서 발급 경로 — 사내 인프라 정책 확인 필요.
- 익명 UUID 정책이 조직의 참여 관리 요구(누가 참여 중인지 파악)와 충돌할 수 있다 — 별칭 필수화 여부를 파일럿에서 결정.
- 팀 구분 단위(부서? 프로젝트?)와 `DailyMetrics`에 팀 필드를 넣을지 — 넣는다면 자유 문자열이 아니라 서버 등록 팀 ID 열거형이어야 함(R2 정합).
- 서버도 이 저장소에 두는 단일 리포 구성이 사내 배포 파이프라인과 맞는가.
