# 스펙: agent-coach v2 — LLM 코칭 개인화·주간 추세·코칭 효과 측정

- 날짜: 2026-07-15 / 상태: 초안
- 관련: `2026-07-14-mvp.md`(선행 — 로드맵 v2 항목), 하네스 도메인 규칙 2·3항(관리 블록 상한, 데이터 반출 금지)

## 배경

MVP의 코칭은 규칙별 고정 템플릿 문장이다. 같은 발견이 반복되면 같은 문장이 반복돼 사용자가 지침을 "배경 소음"으로 학습하고, 사용자 고유의 작업 패턴(자주 쓰는 프로젝트·반복되는 실수 유형)이 문장에 반영되지 않는다. 또한 일일 리포트만 있어 주 단위 추세(개선이 정착됐는가, 특정 요일에 낭비가 몰리는가)를 볼 수 없고, **기입한 지침이 실제로 행동을 바꿨는지**를 측정하지 않아 코칭 루프가 열린 채로 끝난다.

## 목표

- 코칭 문장이 사용자의 실제 발견 데이터에 맞게 개인화된다 — 단, LLM 호출이 불가능·실패·상한 초과인 경우 MVP 템플릿으로 언제나 폴백한다.
- 주 1회 7일 롤업 리포트가 생성되어 규칙별 추이와 요일 패턴을 보여준다.
- 지침 기입 후 해당 신호의 증감을 다음 측정이 기록한다 — "코칭이 효과가 있었는가"가 데이터로 답해진다.
- LLM 사용 비용에 일일 상한이 있고, 상한이 지켜졌음이 로그로 증명된다.

## 비목표

- 세션 원문·프롬프트 내용·파일 경로를 LLM 프롬프트에 넣는 것 — 시크릿·회사 데이터 반출 위험(하네스 도메인 규칙 3항). LLM에는 **집계 수치와 규칙 발견 요약만** 보낸다.
- 대화형 코칭(챗봇) — 이 서비스는 무인 데몬이다. 질문은 리포트가 답한다.
- 관리 블록 15줄 상한의 완화 — 개인화는 문장의 질을 바꾸는 것이지 양을 늘리는 것이 아니다.
- 팀·조직 단위 비교 — v3의 몫.

## 요구사항

- R1 (LLM 엔진 선택): 개인화는 opt-in(`config.json`의 `llm.enabled`, 기본 false)이다. 엔진 우선순위 — ① PATH의 `claude` CLI headless(`claude -p`, 사용자가 이미 인증한 환경 재사용) ② `ANTHROPIC_API_KEY` 환경 변수 기반 직접 호출 ③ 둘 다 없으면 비활성으로 간주하고 템플릿 유지. 이유: 별도 키 발급을 요구하면 비개발자 설치 단순성이 깨진다.
- R2 (프롬프트 데이터 최소화): LLM 프롬프트에는 다음만 허용된다 — 규칙 이름, 발견 수치(건수·점수·추세 %), 지난 7일 규칙별 카운트, 이전 지침의 효과 측정 결과(R6). 세션 원문·first_message·프로젝트명·파일 경로·머신명은 **화이트리스트 밖이므로 코드 구조상 프롬프트 빌더에 전달되지 않아야** 한다(전달 후 필터링이 아니라 타입 수준 차단).
- R3 (출력 계약): LLM 출력은 지침 문장 최대 3개(각 1줄, 160룬 이내)로 강제한다. 형식 위반·초과·빈 출력이면 해당 실행은 템플릿으로 폴백하고 리포트 주석에 남긴다. 관리 블록 15줄 상한(MVP R4)은 불변이다.
- R4 (비용 상한): 일일 LLM 호출은 1회, 프롬프트+출력 합산 토큰 상한을 config(`llm.daily_token_cap`, 기본 4000)로 강제한다. 상한 초과 예상 시 호출하지 않고 템플릿 폴백. 호출 여부·사용 토큰을 state에 기록한다 — 토큰 절감 서비스가 자기 비용을 숨기면 안 된다.
- R5 (주간 리포트): 주 1회(기본 월요일, 일일 analyze 내부에서 판정 — 별도 스케줄 등록 없음) `~/.agentcoach/reports/weekly-YYYY-Www.md`를 생성한다. 내용: 7일 토큰·비용 합계와 전주 대비, 규칙별 발견 추이 표, 요일별 분포, 효과 측정 요약(R6). 데스크톱 알림은 주간 리포트 생성도 알린다.
- R6 (코칭 효과 측정): 지침을 기입한 규칙에 대해, 다음 측정에서 같은 규칙의 신호량(건수·점수)을 직전과 비교해 `improved | unchanged | worsened`로 state에 기록한다. 이 결과는 일일 리포트("어제 지침의 효과")와 주간 리포트, LLM 프롬프트(R2 허용 항목)에 쓰인다. 이유: 효과 없는 지침이 반복 기입되는 것을 감지해야 코칭 루프가 닫힌다.
- R7 (하위 호환): v2 기능 전부가 꺼진 상태(기본값)에서 MVP와 동작이 완전히 같아야 한다. 기존 `config.json`·`state.json`은 마이그레이션 없이 로드된다(누락 필드 = 기본값, MVP config 정책 계승).

## 인터페이스 / 설계 개요

```
internal/llm/        — 엔진 탐지(claude CLI → API 키 → 없음), 프롬프트 빌더(화이트리스트 타입), 호출·상한 집행
internal/effect/     — 규칙별 신호량 전회 대비 비교, improved/unchanged/worsened 판정
internal/report/     — weekly.go 추가 (7일 롤업 렌더)
```

- `config.json` 추가 필드: `llm: {enabled, daily_token_cap}`, `weekly_report: {enabled, weekday}`.
- `state.json` 추가 필드: `rule_history`(규칙별 최근 N회 점수), `llm_usage`(날짜·토큰), `last_weekly_date`.
- 프롬프트 빌더 입력 타입(경계면): `type CoachContext struct { Rules []RuleStat; Trend TrendStat; Effects []EffectStat }` — 문자열 자유 필드 없음(R2의 타입 수준 차단).
- CLI 추가: `agentcoach report --weekly`(최신 주간 리포트 출력). 새 서브커맨드는 없다.

## 완료 기준 (테스트 가능한 형태)

- [ ] C1 (R1): `llm.enabled=false`(기본)면 LLM 코드 경로가 호출되지 않고 MVP 템플릿 문장이 기입된다. 엔진 부재 시 enabled여도 템플릿 폴백 + 리포트 주석.
- [ ] C2 (R2): 프롬프트 빌더에 세션 원문·프로젝트명을 전달하는 코드 경로가 컴파일 수준에서 존재하지 않는다 — `CoachContext`에 해당 필드가 없음을 타입으로 확인하고, 생성된 프롬프트 문자열에 픽스처의 세션 원문 조각이 포함되지 않음을 테스트.
- [ ] C3 (R3): LLM 출력 픽스처 4종(정상 / 4개 초과 / 줄바꿈 포함 / 빈 출력)에서 정상만 채택되고 나머지는 템플릿 폴백. 블록은 어느 경우에도 15줄 이내.
- [ ] C4 (R4): 상한 초과 예상 픽스처에서 호출 미발생 + state의 `llm_usage`에 기록. 일일 2회째 analyze(--force)는 LLM을 재호출하지 않는다.
- [ ] C5 (R5): 시각 고정 테스트에서 월요일 실행 시 주간 리포트가 생성되고 화요일~일요일에는 생성되지 않는다. 7일 합계·전주 대비 수치가 픽스처와 일치.
- [ ] C6 (R6): 같은 규칙의 점수가 감소한 픽스처 → `improved`, 증가 → `worsened`가 state에 기록되고 다음 일일 리포트에 표시된다.
- [ ] C7 (R7): v2 필드가 없는 MVP 시절 `config.json`·`state.json`을 로드해도 에러 없이 기본값으로 동작한다.

## 미해결 질문

- `claude -p` 호출도 결국 외부 API 전송이다 — 집계 수치만 보내더라도 사내 보안 정책상 외부 LLM 호출 자체의 승인이 필요한가? (사내 배포 전 보안 검토 필요. 불허 시 R1의 엔진 목록에 사내 LLM 게이트웨이를 추가하는 변형)
- 주간 리포트 기본 요일(월요일)이 적절한가 — 사내 파일럿 피드백으로 확정.
